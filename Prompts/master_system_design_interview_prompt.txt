
Below is the prompt I'd recommend using as your master system design interview prompt. 
The biggest change I made is that instead of having a special section (say only for Payment Systems), 
it tells the model to automatically infer the domain (Payment, Uber, YouTube, Dropbox, Kafka, etc.) 
and include everything expected for that domain. That way you can reuse the exact same prompt for any
interview question.




# Staff Software Engineer System Design Interview Prompt

Act as a Staff Software Engineer preparing for a 30-minute system design interview.

Design the following distributed system:

# [SYSTEM NAME HERE]

---

# Goal

Produce a **single, complete, interview-ready study note** that I can use to prepare for a Staff/Principal Software Engineer system design interview.

The output should read like the notes of an experienced engineer rather than a textbook.

Do **NOT** skip any section.

Do **NOT** generate multiple versions.

Do **NOT** repeat sections.

Everything should appear exactly once.

---

Instead include everything an experienced interviewer expects specifically for that domain, including:

# General Design Principles

Every major component/service must explain:

- What problem it solves
- What happens if we remove it
- Common failure modes
- Scaling considerations
- Interview explanation line

For every component include an interview script:

> "..."

that I can literally speak while drawing.

# Formatting Rules

- Use Markdown.
- Use Mermaid flowchart LR.
- Wrap every Mermaid node label in double quotes.
- Number every arrow.
- Use blockquotes for interview scripts.
- Use tables whenever appropriate.
- Do not use colors.
- Do not generate images.
- Avoid redundancy.
- Keep explanations concise but interview-ready.
- The final output should be one continuous document.

---

# Quality Bar

Assume the interviewer is a Staff or Principal Engineer from companies such as 
Google, Meta, Stripe, Uber, Airbnb, LinkedIn, DoorDash, Amazon, Databricks, Snowflake, 
OpenAI, Anthropic, or Netflix.

The design should be production-ready, technically accurate, explain trade-offs clearly, 
and include the domain-specific depth expected in a real Staff-level system design interview.
